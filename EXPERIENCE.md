# Spoken 项目开发经验总结

> Last updated: 2026-04-06

---

## 背景

Spoken 是一个 macOS 菜单栏语音输入工具，核心技术链：
```
麦克风 → 语音识别 → AI优化 → 文本注入到焦点窗口
```

开发过程中参考了开源项目 type4me（github.com/joewongjc/type4me），它完美跑通了全流程，而 spoken 始终无法稳定注入文本。

---

## 一、核心教训

### 1. 先研究，再动手

**教训：做任何技术方案之前，先去找同类产品的现成实现研究清楚。不要闭门造车，不要觉得自己的想法比现成方案更周全。**

这次做 spoken，我拿到需求就自己琢磨技术路线，完全没有先去看看有没有现成实现。如果一开始就 clone type4me 源码读一遍，很多问题根本不会发生。

**正确姿势：**
- 遇到技术难题 → 先搜索 GitHub 有没有类似实现
- 找到参考项目 → clone 下来仔细研究
- 理解原理 → 再动手自己的方案

### 2. 基础 API 要查文档，不要想当然

**教训：不要假设自己知道某个 API 的行为，要查文档确认。**

最典型的例子：`usleep(50)` — 我以为参数是毫秒，实际是**微秒**，差了 1000 倍。

- `usleep(50)` = 等待 0.05 毫秒（几乎不等）
- `usleep(50_000)` = 等待 50 毫秒

这导致 spoken 的文本注入：写完剪贴板后几乎立刻就粘贴，目标 App 还没读到新内容。

**正确姿势：**
- 用一个不熟悉的 API 前，查 Apple Developer Documentation
- 关键参数写完就验证，不要等到整体联调才发现

### 3. 逐个模块验证，不要整体跑一遍了事

**教训：每个关键函数要有独立的输入输出验证，不能等到整体跑起来才发现问题。**

spoken 里的 `sendAudio(_ data: Data)` 是空实现，这个函数从来没被调用过，音频数据其实从来没传进识别引擎。但因为没有单独测试，一直不知道这个问题存在。

**正确姿势：**
- 每个模块单独验证：录音能不能用 → 识别能不能用 → 注入能不能用
- 发现问题就定位到具体函数，不要整体猜测

### 4. 遇到卡点立刻研究参考实现，不要自己猜着改

**教训：遇到自己解决不了的问题，第一反应应该是去找现成方案，而不是反复试错。**

spoken 的文本注入功能前后改了无数次，每次都是猜着改。应该直接 clone type4me 对比差异，而不是凭猜测尝试。

## 二、技术细节教训

### usleep 单位陷阱

```swift
// ❌ 错误 — 50 微秒，几乎不等
usleep(50)

// ✅ 正确 — 50 毫秒
usleep(50_000)
```

### SFSpeechRecognizer 音频传递

```swift
// ❌ 错误 — sendAudio 是空实现，音频没传进去
try await client.sendAudio(data)

// ✅ 正确 — 直接 append PCM buffer
request.append(buffer)
```

### 文本注入时序

| 步骤 | spoken（错误） | type4me（正确） |
|------|--------------|----------------|
| 写剪贴板 | ✓ | ✓ |
| 等待 | 0.05ms | 50ms |
| 模拟粘贴 | ✓ | ✓ |
| 等待 | 0.1ms | 100ms |
| 恢复剪贴板 | 无 | ✓ |

## 三、为什么 type4me 完美跑通

type4me 的 RecognitionSession 有 1125 行代码，每个环节都有：

1. **状态机**：6种状态（idle/starting/recording/finishing/injecting/postProcessing）
2. **超时兜底**：每个 await 都有 timeout，有 forceReset 机制
3. **Batch Fallback**：streaming 断了用完整音频重发一次
4. **Task.detached 注入**：不阻塞主 actor
5. **Speculative LLM**：录音时预热 LLM，减少感知延迟

spoken 的实现不是思路错了，是边角情况和异常处理太薄。

## 四、方法论总结

**开始任何技术项目之前，按这个顺序：**

1. **搜索现成方案** — GitHub、App Store 类似产品
2. **研究参考实现** — clone 下来仔细读
3. **制定技术方案** — 基于研究结果，不是凭空想象
4. **逐模块验证** — 每个模块单独测试
5. **对比找差异** — 遇到问题，对照参考实现

**这是避免重复造轮子、避免基础错误的唯一有效方法。**
